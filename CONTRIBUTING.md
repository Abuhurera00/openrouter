# Contributing to OpenRouter

Thank you for your interest in contributing to OpenRouter! We appreciate your help in making this project better. This document provides guidelines and instructions for contributing.

## Code of Conduct

We are committed to providing a welcoming and inspiring community. Please read and follow our community guidelines to help us maintain a positive environment for all contributors.

## Getting Started

### Prerequisites

- Node.js 18 or higher
- pnpm 9.0.0 or higher
- Git
- MongoDB (for local development)

### Development Setup

1. **Fork the repository**
   ```bash
   git clone https://github.com/yourusername/openrouter.git
   cd openrouter
   ```

2. **Install dependencies**
   ```bash
   pnpm install
   ```

3. **Set up environment variables**
   ```bash
   cp apps/primary-backend/.env.example apps/primary-backend/.env.local
   cp apps/api-backend/.env.example apps/api-backend/.env.local
   cp apps/web/.env.example apps/web/.env.local
   ```

4. **Start development servers**
   ```bash\n   pnpm dev\n   ```\n\n5. **Verify setup**\n   ```bash\n   pnpm lint\n   pnpm check-types\n   pnpm test\n   ```\n\n## How to Contribute\n\n### 1. Finding Issues to Work On\n\n- Check the [Issues](https://github.com/yourusername/openrouter/issues) page\n- Look for issues labeled `good-first-issue` or `help-wanted`\n- Feel free to open a new issue if you find a bug or have a feature request\n\n### 2. Creating a Feature Branch\n\n```bash\n# Update main branch\ngit checkout main\ngit pull origin main\n\n# Create a new branch\ngit checkout -b feature/your-feature-name\n# or\ngit checkout -b fix/your-bug-fix-name\n```\n\n**Branch naming convention:**\n- `feature/short-description` - for new features\n- `fix/short-description` - for bug fixes\n- `refactor/short-description` - for code refactoring\n- `docs/short-description` - for documentation\n- `test/short-description` - for test additions\n\n### 3. Making Changes\n\n#### Code Style\n\nWe follow strict code style guidelines to maintain consistency:\n\n- **TypeScript**: Use strict type checking\n- **Formatting**: Prettier (automatic via git hooks)\n- **Linting**: ESLint rules as defined in the repo\n- **Naming**: camelCase for variables/functions, PascalCase for classes/types\n\n#### Running Validation\n\n```bash\n# Format your code\npnpm format\n\n# Lint your code\npnpm lint\n\n# Check types\npnpm check-types\n\n# Run tests\npnpm test\n```\n\n#### File Organization\n\n```typescript\n// Import order:\n// 1. External dependencies\n// 2. @nestjs modules\n// 3. Local imports\n// 4. Relative imports\n\nimport { Controller } from '@nestjs/common';\nimport { MyService } from './my.service';\n```\n\n#### TypeScript Best Practices\n\n```typescript\n// ✅ DO: Use explicit types\nfunction calculateTotal(price: number, tax: number): number {\n  return price + tax;\n}\n\n// ❌ DON'T: Avoid 'any'\nfunction calculateTotal(price: any, tax: any): any {\n  return price + tax;\n}\n\n// ✅ DO: Use interfaces for contracts\ninterface User {\n  id: string;\n  email: string;\n  name: string;\n}\n\n// ✅ DO: Use enums for constants\nenum UserRole {\n  ADMIN = 'ADMIN',\n  USER = 'USER',\n}\n```\n\n### 4. Writing Tests\n\nNew features must include tests. We use Jest for testing.\n\n```typescript\n// Example: my-feature.spec.ts\ndescribe('MyFeature', () => {\n  let service: MyService;\n\n  beforeEach(async () => {\n    const module = await Test.createTestingModule({\n      providers: [MyService],\n    }).compile();\n    service = module.get<MyService>(MyService);\n  });\n\n  describe('someMethod', () => {\n    it('should return expected result', () => {\n      const result = service.someMethod();\n      expect(result).toBe(true);\n    });\n\n    it('should handle edge cases', () => {\n      const result = service.someMethod(null);\n      expect(result).toBeFalsy();\n    });\n  });\n});\n```\n\n**Test coverage requirements:**\n- Minimum 80% line coverage for new code\n- All public APIs must be tested\n- Happy path and error cases must be covered\n\nRun tests:\n```bash\npnpm test\npnpm test:watch\npnpm test:cov\n```\n\n### 5. Commit Guidelines\n\nWe follow conventional commits for clear changelog generation.\n\n**Format:**\n```\ntype(scope): subject\n\nbody\n\nfooter\n```\n\n**Types:**\n- `feat`: A new feature\n- `fix`: A bug fix\n- `docs`: Documentation changes\n- `style`: Code style changes (formatting, missing semicolons, etc.)\n- `refactor`: Code refactoring\n- `perf`: Performance improvements\n- `test`: Test additions or modifications\n- `chore`: Dependency updates, build configuration\n\n**Examples:**\n```bash\nfeat(auth): add oauth2 support\nfix(chat): handle streaming timeout correctly\ndocs(readme): update installation instructions\nrefactor(llm): extract provider interface\ntest(api): add integration tests for endpoints\n```\n\n**Commit message best practices:**\n- Use imperative mood (\"add\" not \"added\")\n- Don't capitalize the subject\n- Keep subject under 50 characters\n- Reference issues in the footer: `Closes #123`\n- Provide detailed description in the body\n\n### 6. Creating a Pull Request\n\n1. **Push your branch**\n   ```bash\n   git push origin feature/your-feature-name\n   ```\n\n2. **Create a Pull Request on GitHub**\n   - Use a descriptive title\n   - Fill in the PR template completely\n   - Link related issues using `Closes #123`\n   - Add screenshots/videos if applicable\n\n3. **PR Template**\n   ```markdown\n   ## Description\n   Brief description of changes\n\n   ## Type of Change\n   - [ ] Bug fix\n   - [ ] New feature\n   - [ ] Breaking change\n   - [ ] Documentation update\n\n   ## Related Issues\n   Closes #123\n\n   ## Testing\n   Describe how to test these changes\n\n   ## Checklist\n   - [ ] Code follows style guidelines\n   - [ ] Self-review completed\n   - [ ] Comments added for complex logic\n   - [ ] Documentation updated\n   - [ ] Tests added/updated\n   - [ ] All tests pass\n   ```\n\n### 7. Code Review Process\n\n- At least one approval is required before merging\n- Address reviewer feedback promptly\n- Keep discussions professional and respectful\n- Ask questions if feedback is unclear\n- Request re-review after making changes\n\n## Project Structure\n\n### Adding Features\n\n#### To Primary Backend:\n```\napps/primary-backend/src/\n├── feature-name/\n│   ├── feature-name.controller.ts\n│   ├── feature-name.service.ts\n│   ├── feature-name.module.ts\n│   ├── dto/\n│   │   ├── create-feature.dto.ts\n│   │   └── update-feature.dto.ts\n│   ├── entities/\n│   │   └── feature-name.entity.ts\n│   ├── feature-name.controller.spec.ts\n│   └── feature-name.service.spec.ts\n```\n\n#### To API Backend:\n```\napps/api-backend/src/\n├── feature-name/\n│   ├── feature-name.controller.ts\n│   ├── feature-name.service.ts\n│   ├── feature-name.module.ts\n│   └── feature-name.service.spec.ts\n```\n\n#### To Web Frontend:\n```\napps/web/src/\n├── components/FeatureName/\n│   ├── FeatureName.tsx\n│   ├── FeatureName.module.css\n│   └── FeatureName.test.tsx\n├── pages/feature-page/\n│   └── page.tsx\n└── hooks/useFeatureName.ts\n```\n\n## Documentation\n\n- Update README.md if adding user-facing features\n- Add JSDoc comments to exported functions/classes\n- Update API documentation for new endpoints\n- Include examples for complex features\n\n```typescript\n/**\n * Fetches user data by ID\n * @param userId - The unique identifier of the user\n * @returns Promise containing user data\n * @throws Error if user not found\n * @example\n * const user = await userService.findById('123');\n */\nasync findById(userId: string): Promise<User> {\n  // implementation\n}\n```\n\n## Performance Considerations\n\n- Avoid N+1 queries in database operations\n- Use caching for frequently accessed data\n- Implement pagination for large datasets\n- Optimize frontend re-renders\n- Use appropriate indexes in MongoDB\n- Profile before and after performance changes\n\n## Security Guidelines\n\n- Never commit secrets or API keys\n- Validate all user inputs\n- Use parameterized queries (Mongoose handles this)\n- Implement proper authentication/authorization\n- Follow OWASP guidelines\n- Report security issues privately to maintainers\n\n## Common Tasks\n\n### Adding a New NestJS Module\n\n1. Create the module structure\n2. Implement controller, service, and DTOs\n3. Add tests\n4. Import in the appropriate parent module\n5. Update API documentation\n\n### Adding a New Database Schema\n\n1. Create schema in `packages/database/src/schemas/`\n2. Create repository in `packages/database/src/repositories/`\n3. Export from `packages/database/src/index.ts`\n4. Add to database module\n\n### Updating Dependencies\n\n```bash\n# Check for updates\npnpm update --latest\n\n# Update specific package\npnpm update package-name@latest\n\n# Update all in workspace\npnpm -r update\n```\n\n## Troubleshooting\n\n### Build Errors\n\n```bash\n# Clear cache and reinstall\nrm -rf node_modules\npnpm install\n\n# Rebuild all\npnpm build --force\n```\n\n### Test Failures\n\n```bash\n# Run specific test file\npnpm test feature.spec.ts\n\n# Run with debugging\nnode --inspect-brk ./node_modules/.bin/jest --runInBand\n```\n\n### Type Errors\n\n```bash\n# Check types across all packages\npnpm check-types\n\n# Fix linting issues\npnpm lint --fix\n```\n\n## Release Process\n\n1. Update version in root package.json\n2. Update CHANGELOG.md\n3. Create a new git tag\n4. Push to repository\n5. Create GitHub release with notes\n6. Deploy to production\n\n## Need Help?\n\n- 📖 Check [documentation](docs/)\n- 💬 Open a [discussion](https://github.com/yourusername/openrouter/discussions)\n- 🐛 Report bugs in [issues](https://github.com/yourusername/openrouter/issues)\n- 📧 Email: dev@openrouter.dev\n\n## Recognition\n\nAll contributors are recognized in our [CONTRIBUTORS.md](CONTRIBUTORS.md) file and receive credit in release notes.\n\n## License\n\nBy contributing to OpenRouter, you agree that your contributions will be licensed under the MIT License.\n\n---\n\nThank you for contributing to OpenRouter! 🚀\n
